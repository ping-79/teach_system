# 教学管理系统 · 总体蓝图与路线图

> 版本：v1（2026-06-16）
> 目标：把现有的"教学进度表"和"教案生成"两个模块，扩展成一套覆盖日常教学管理工作的统一系统；
> 后续可持续加模块，且不影响已上线功能的正常运行。
> 部署：一台较老的电脑（Windows Server），接入 DeepSeek 大模型辅助日常工作。

---

## 一、核心结论

**不需要重做系统。** 现有 `main-app` 已经是一套成型的"统一门户 + 数据中心"，新模块以"加表 + 路由 + 页面"的方式挂上去即可，彼此独立、互不影响。`jiaoan`（教案）这类独立子系统继续用"挂接（iframe / 反代）"的方式接入门户。

| 现状 | 说明 |
| --- | --- |
| `main-app`（端口 3001） | Node/Express + EJS + Prisma + SQLite。已有：登录、管理员/教师权限、学期管理、课表 Excel 导入、**教学进度表**自动生成+编辑+导出 Word、实践进度表、课程内容库、DeepSeek（内容解析 + 进度表 AI 审核）、审批流。首页 `src/views/dashboard/modules.ejs` 即模块导航页，已预留"教师档案"入口。 |
| `jiaoan`（端口 3100，`新建文件夹`为其副本） | NestJS + Python 的**教案生成**系统，DeepSeek 出教案 + python-docx 套模板，已通过 iframe 挂在门户里。 |

---

## 二、总体架构（分层）

```
┌─────────────────────────────────────────────────────────────┐
│  统一门户（main-app）：登录 / 权限（管理员·教师）/ 模块导航      │
├─────────────────────────────────────────────────────────────┤
│  横向能力层（全系统共用）                                       │
│   · DeepSeek AI 辅助   · Word/Excel 导出   · 审批工作流          │
│   · 通知/待办   · 操作审计日志   · 数据备份                      │
├─────────────────────────────────────────────────────────────┤
│  基础数据层（主数据，多模块依赖）                                │
│   院系/教研室 · 专业 · 班级 · 课程库 · 教室 · 教师账号           │
├─────────────────────────────────────────────────────────────┤
│  业务模块层                                                     │
│   [已建成] 教学进度表 · 教案生成 · 课程内容库                    │
│   [规划]  教师档案+业绩 · 教师统计图 · 教材选用+推荐             │
│           试题库+试卷审核 · 排课系统                            │
│   [可选]  教学大纲/课标 · 听课评课 · 教研活动 · 学生评教 · 看板  │
└─────────────────────────────────────────────────────────────┘
```

**扩展约定（保证"加需求不影响已有功能"）**
1. 每个新模块 = 独立的 Prisma 模型 + 独立的 `src/routes/*.js` + 独立的 `src/views/*` + （可选）`src/services/*`。
2. 数据库变更走 Prisma 迁移（新增表/字段，不动旧表语义）。
3. 所有 AI 功能保留"无 AI 也能用"的降级（沿用现有 `deepseek.js` + 启发式回退的写法）。
4. 复用已有能力：`services/word-export.js`（模板导出）、`services/deepseek.js`、审批流（`LibraryUpdateRequest` 模式）、`middleware/auth.js`（权限）。

## 三、基础数据层（主数据）

排课、教材、试题、统计等多个模块都依赖它，建议尽早统一维护，避免每个模块各存一份导致数据不一致。

| 主数据 | 关键字段 | 来源 |
| --- | --- | --- |
| 院系/教研室 | 名称、负责人 | 手工维护 |
| 专业 | 专业名、所属院系、学制 | 手工维护 |
| 班级 | 班级名、专业、年级、人数 | 手工 / 课表导入 |
| 课程库 | 课程名、归一化名、课程性质、总学时 | 手工 / 课表导入（现已部分derive） |
| 教室 | 教室号、类型(理论/实训)、容量、校区 | 手工维护 |
| 教师账号 | 工号、姓名、教研室、专业（见模块一扩展） | 现有 `TeacherProfile` |

> 落地策略：不必一次做全。先随"教师档案"模块补齐教师主数据；专业/班级/教室/课程随"排课"模块再完整化。

## 四、各业务模块蓝图

每个模块统一格式：**作用 → 数据模型要点 → DeepSeek 作用 → 导出 → 依赖**。

### 模块 1 ·【优先】教师个人信息 + 业绩材料档案
> 详细设计见《教师档案与业绩模块-详细设计.md》。核心思路：**把"职称/岗位申报"反过来变成"日常"**——平时按类目随手录业绩、传佐证扫描件，系统自动按《量化评分表》算分，申报季一键导出《竞聘自评（复核）表》《量化评分表》《佐证材料汇编册》。

- **作用**：维护教师基础信息；按年度结构化归档业绩；自动量化算分；多级复核；一键生成职称/岗位申报材料。
- **数据来源**：已读取 `00相关资料/` 四份文件，业绩类目与算分规则直接对齐《教师系列职称申报量化评分表》《竞聘自评（复核）表》，佐证编目对齐《佐证材料汇编册》。
- **数据模型要点**：
  - 扩展 `TeacherProfile`：性别、出生年月、政治面貌、最高学历/学位、职称+岗位等级+评定时间、职务、教研室、所属专业、**参加工作时间 / 来校时间（分别算工龄、校龄）**、联系方式、照片、在职状态。
  - **统一 `Achievement` 表**（不再分 7 张表）：`category`(资历经历荣誉/教学成果/科研业绩) + 子类 + `level`(国/省/市厅/校) + `rank`(特/一/二/三等) + `authorOrder`(本人排序→系数) + 日期 + 自评描述 + `detailJson`(各类型专属字段) + 自评/初复/终复得分 + 复核部门 + 状态。
  - `AchievementAttachment` 佐证扫描件（含 OCR 文本与识别状态）。
  - **配置化评分方案 `ScoringScheme`/`ScoringItem`**：把"级别×等次→分值""排序系数""就高不就低""分项封顶/限分组"等规则存成数据，政策变了改配置即可、不改代码。
  - `TitleApplication` 申报/竞聘批次：归集某年度业绩 + 总分 + 提交快照(snapshotJson) + 状态机。
- **AI / OCR**：上传佐证扫描件 → OCR/多模态识别 → 预填字段 → **人工确认**（草稿确认流，复用内容库模式）；可生成"年度业绩小结"。所有 AI 功能保留无 AI 降级。
- **导出**：《竞聘自评（复核）表》《量化评分表》《佐证材料汇编册（目录+排序）》《个人业绩档案》，复用 `word-export.js` 模板套打，模板取自那四份文件。
- **复核工作流**：自评得分 → 初步复核 → 最终复核，标注负责复核部门（组织人事处/教务处/科研处/学生处…），对齐自评表的列。
- **依赖**：无（是其它模块的数据基础）。门户已有占位入口 `/teacher-profile`。约 100 名教师自助登录录入。

### 模块 2 · 教师基本资料统计图
- **作用**：全体教师结构画像，供领导/评估使用。
- **图表**：学历分布、职称分布、年龄分布、学缘结构、专业/教研室分布、双师型比例、业绩数量趋势。
- **技术**：ECharts 或 Chart.js（纯前端、离线可用，老电脑无压力）。
- **导出**：统计报表 Word/Excel。
- **依赖**：⚠️ 依赖【模块 1】的教师与业绩数据，因此排在模块 1 之后。

### 模块 3 · 教材选用 + 教材推荐表
- **作用**：维护教材库，按课程/学期登记选用，生成教材推荐/选用审批表。
- **数据模型要点**：`Textbook`(书名、ISBN、作者、出版社、版次、出版日期、是否国家规划/马工程教材、价格)、`TextbookSelection`(课程、学期、选用理由)、`TextbookRecommendation`(推荐表 + 审批状态)。
- **DeepSeek 作用**：按书名/ISBN 补全校验教材信息、生成"选用理由"。
- **导出**：标准"教材选用审批表"Word。复用审批流。
- **依赖**：课程库（基础数据）。工作量最小、最快见效。

### 模块 4 · 课程试题库 + 试卷审核表
- **作用**：分课程维护题库，组卷，生成试卷与试卷审核表。
- **数据模型要点**：`Question`(课程、题型[单选/多选/判断/填空/简答/计算/案例]、题干、选项、答案、分值、难度、知识点)、`ExamPaper`(组卷)、`ExamPaperItem`、`ExamReview`(审核人、状态、意见)。
- **DeepSeek 作用**：① 从"课程内容库"自动出题；② 自动标难度/知识点；③ 组卷后自动审核（题型/难度/知识点覆盖、查重、总分校验）。
- **导出**：试卷 Word/Excel、试卷审核表。
- **依赖**：课程库、可与已有"课程内容库"打通。

### 模块 5 ·【最后做】各专业各学期排课系统
- **作用**：按专业/学期编排课程，自动检测冲突，输出课表。
- **数据模型要点**：`TeachingTask`(专业+课程+班级+教师+学时)、`ScheduleSlot`(周次、星期、节次、教室)。
- **关键点**：核心是**约束求解算法**（教师/教室/班级冲突、教室容量），用启发式（贪心 + 回溯）实现，**不该交给大模型排**；大模型只用于"解释冲突、给调整建议"。
- **价值延伸**：排课结果可**反向替代**当前"老师手动上传课表 Excel"那一步，与教学进度表自动打通。
- **依赖**：完整的基础数据层（专业/班级/课程/教室/教研室）。最复杂，放最后。

### 可选补充模块（按需）
教学大纲/课程标准管理（进度表、试题、教材的源头）、听课评课记录、教研活动记录、学生评教、跨模块数据看板。

## 七、部署与运维（这台老电脑 / Windows Server）

| 事项 | 建议 |
| --- | --- |
| 数据库 | **SQLite 够用**（一个系/教研室、几十位老师）。仅当全校上百人并发写入才考虑迁 PostgreSQL。 |
| 进程托管 | 用 **nssm** 把 `main-app`(3001) 与 `jiaoan`(3100) 注册成 Windows 服务，开机自启、崩溃自拉起。 |
| 备份 | **每天定时**复制 `data/teaching_materials.db` + `data/uploads` + `data/exports`；保留多份历史。 |
| 反向代理 | 前置反代统一对外域名/端口（你已有 NAS 反代截图），并启用 HTTPS。 |
| DeepSeek | 成本低；但试题、教师个人信息会上云，需明确哪些数据可外发；保留无 AI 降级。 |
| 性能 | 老机器内存优先够用；Node 单进程足以应付小规模；大文件解析（docx/xlsx）注意超时。 |

## 八、关键技术取舍与风险

1. **统一在 main-app 扩展，而非新建项目** —— 复用登录/权限/导出/审批/AI，开发最快、最稳。
2. **排课用算法不用大模型** —— 排课是约束求解，LLM 不可靠；LLM 只辅助解释与建议。
3. **主数据要早规划** —— 否则各模块各存一份，后期清洗成本高。
4. **隐私边界** —— 教师个人信息、试题外发云端 LLM 前要过滤/脱敏。
5. **单机单点风险** —— 老电脑是单点，备份 + 服务自启是底线；预算允许可加一块独立备份盘。
