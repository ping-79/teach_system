# 教师档案与业绩模块 · 详细设计

> 版本：v1（2026-06-16）｜ 所属：教学管理系统 阶段 1 起步模块
> 实现位置：在现有 `main-app`（Express + EJS + Prisma + SQLite）内扩展，不新建项目。
> 设计依据：`00相关资料/` 四份文件——《教师系列职称申报量化评分表》《专业技术岗位竞聘申报表 / 竞聘自评（复核）表 / 师德师风综合考评表》《岗位聘任申报佐证材料》《申报材料汇编册》。

---

## 一、目标与核心思路

教师业绩"平时收得乱"的根因是：**平时不录、申报时才一次性扒材料**。本模块把它反过来——

> **平时按类目随手录业绩 + 传佐证扫描件 → 系统按《量化评分表》自动算分 → 申报季一键导出《竞聘自评（复核）表》《量化评分表》《佐证材料汇编册》。**

三个能力，全部对齐你的真实表格：

1. **自动量化算分**：每条业绩带 `级别 + 等次 + 本人排序`，按评分细则自动算分（含排序系数、就高不就低、分项封顶/限分组）。
2. **多级复核**：自评得分 → 初步复核 → 最终复核，标注负责复核部门（组织人事处 / 教务处 / 科研处 / 学生处…）。
3. **一键出申报材料**：自评表、量化评分表、佐证材料汇编册（自动编目录+排序）。

面向约 **100 名教师自助登录录入**，管理员/部门复核。

## 二、业绩类目（对齐量化评分表）

统一三大类，子类作为 `subType`（不再拆成 7 张表）：

| 大类 `category` | 子类 `subType`（示例） |
| --- | --- |
| 个人资历经历荣誉 `RESUME` | 工龄校龄、多岗兼任、辅导员班主任经历、双师型证书、荣誉奖励、年度考核、乡村振兴/援疆 |
| 教学成果 `TEACHING` | 教学工作量、教学质量评价、教师教学竞赛、指导学生竞赛、课程建设(一流/精品在线)、专业教学资源库、教学创新团队、现代学徒制、1+X、教学成果奖 |
| 科研业绩 `RESEARCH` | 课题、科研奖励、论文、论著/教材教参、知识产权(专利/标准/软著) |

> 工龄/校龄、教学工作量可由系统**自动带出**（来自 `TeacherProfile` 的工作/来校时间、排课课表的周课时），减少手填。

### 3.3 Achievement 业绩（统一表 + detailJson）

```prisma
enum AchievementCategory { RESUME TEACHING RESEARCH }
enum AchievementLevel    { NATIONAL PROVINCIAL MUNICIPAL SCHOOL NONE } // 国/省/市厅/校/无
enum AchievementRank     { SPECIAL FIRST SECOND THIRD NONE }           // 特/一/二/三等/无
enum AchievementReviewStatus { DRAFT SUBMITTED PRELIM_REVIEWED FINAL_REVIEWED REJECTED }

model Achievement {
  id            Int      @id @default(autoincrement())
  teacherId     Int
  year          Int                  // 业绩年度（按年归档）
  category      AchievementCategory
  subType       String               // 子类（见第二节）
  scoringItemId Int?                 // 命中的评分细则项（用于自动算分）

  title         String               // 业绩名称/题目
  level         AchievementLevel @default(NONE)
  rank          AchievementRank  @default(NONE)
  rankLabel     String?              // 原始等次文字（如"一等奖"）
  authorOrder   Int?                 // 本人排序（→系数）
  authorRole    String?              // 主持/第X参与人/通讯作者…
  happenedOn    DateTime?            // 取得/发表/授权/结题日期
  selfDescription String?            // 自评描述

  detailJson    String?              // 类型专属字段（期刊/检索/专利号/课题来源经费/班级课时…）

  computedScore Float?               // 系统按规则自动算
  selfScore     Float?               // 自评得分
  prelimScore   Float?               // 初步复核得分
  finalScore    Float?               // 最终复核得分
  reviewDept    String?              // 负责复核部门
  reviewStatus  AchievementReviewStatus @default(DRAFT)
  reviewNote    String?
  verified      Boolean  @default(false)  // 已认定（盖"印章"徽章）

  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt
  teacher     TeacherProfile @relation(fields: [teacherId], references: [id], onDelete: Cascade)
  scoringItem ScoringItem?   @relation(fields: [scoringItemId], references: [id], onDelete: SetNull)
  attachments AchievementAttachment[]

  @@index([teacherId, year, category])
}
```

`detailJson` 各子类约定（示例）：
- 论文：`{ journal, issn, indexed:"核心/CSSCI/一般", page, isCorresponding }`
- 课题：`{ source, fundingAmount, startOn, endOn, status:"立项/结题" }`
- 专利：`{ patentType:"发明/实用新型/外观", patentNo }`
- 竞赛：`{ category:"国一类/国二类/省一类…", isStudentGuide, sameItemTeacherCount }`
- 课时量：`{ semesterId, weeklyPeriods }`（可由排课/课表自动带出）

### 3.5 配置化评分方案（规则存数据，政策变只改配置）

```prisma
enum ScoringApplyType { TITLE POST }            // 职称申报 / 岗位竞聘
enum ScoringScoreType { FIXED PER_UNIT LOOKUP } // 固定分 / 按年按项 / 按级别等次查表

model ScoringScheme {
  id        Int      @id @default(autoincrement())
  name      String   // 如"2024 职称申报量化评分办法"
  applyType ScoringApplyType
  policyNo  String?  // 文号，如 赣建院党字〔2024〕13号
  year      Int?
  isActive  Boolean  @default(true)
  items     ScoringItem[]
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
}

model ScoringItem {
  id           Int      @id @default(autoincrement())
  schemeId     Int
  category     AchievementCategory
  code         String   // 评分表编号，如 2-6
  name         String   // 评分项名称
  capGroupCode String?  // 限分组（如"教学成果限50"）
  maxScore     Float?   // 该项封顶
  scoreType    ScoringScoreType
  configJson   String   // 级别×等次→分值表、排序系数表、就高不就低、按年单位分值…
  createdAt    DateTime @default(now())
  updatedAt    DateTime @updatedAt
  scheme       ScoringScheme @relation(fields: [schemeId], references: [id], onDelete: Cascade)
  achievements Achievement[]

  @@unique([schemeId, code])
}
```

### 3.6 TitleApplication 申报/竞聘批次

```prisma
enum ApplicationStatus { DRAFT SUBMITTED REVIEWING APPROVED RETURNED }

model TitleApplication {
  id             Int      @id @default(autoincrement())
  teacherId      Int
  schemeId       Int
  applyType      ScoringApplyType
  applyLevel     String?  // 申报级别，如"高职副教授"
  year           Int
  status         ApplicationStatus @default(DRAFT)
  totalSelfScore Float?
  totalFinalScore Float?
  snapshotJson   String?  // 提交时冻结的业绩+得分快照（后续改业绩不影响已交批次）
  submittedAt    DateTime?
  reviewedAt     DateTime?
  createdAt      DateTime @default(now())
  updatedAt      DateTime @updatedAt
  teacher        TeacherProfile @relation(fields: [teacherId], references: [id], onDelete: Cascade)
}
```

> 业绩纳入批次的方式：按 `teacherId + 年度/年限范围`自动归集 + 允许勾选取舍；提交时写入 `snapshotJson` 冻结（沿用现有 `LibraryUpdateRequest.snapshotJson` 模式）。

## 四、算分引擎

新增 `src/services/scoring.js`：

- `scoreAchievement(achievement, scoringItem)` → 读取 `scoringItem.configJson` 的"级别×等次→分值"表，乘以 `authorOrder` 对应的排序系数，返回 `computedScore`。
- `scoreApplication(application)` → 汇总该批次所有业绩：先按"细项就高不就低不累计"取最高，再按"限分组封顶"截断（如教学成果限 50、个人经历限 20），输出三大类小计与总分。
- 规则全部来自配置，代码只做"查表 × 系数 → 取最高 → 封顶"的通用计算。
- 排序系数等规则在 `configJson` 里声明，例如：
  `{ "levelRank": {"PROVINCIAL":{"FIRST":9,"SECOND":7,"THIRD":5}}, "orderCoef":[1,0.75,0.5,0.375,0.3], "orderCoefRest":0.2, "pickMaxOnly":true }`

> 分期：**v1** 可先支持"系统给建议分 + 老师自评分可改"；评分方案配置后再开启全自动算分（见第十节）。

## 五、OCR / AI 预填（草稿确认流，复用内容库模式）

新增 `src/services/ocr.js`（可插拔 provider）+ `src/services/achievement-ai.js`：

流程：**上传扫描件 → OCR/多模态识别文本 → DeepSeek 结构化出字段 → 预填表单(高亮可改) → 老师确认入库**。绝不盲信识别结果；原始扫描件始终留存为佐证。

三条可选技术路线（默认①，env 可切换）：
1. **云 OCR + DeepSeek 结构化（推荐）**：百度/腾讯/阿里通用文字识别取文本 → `deepseek.js` 出 JSON。老电脑零负担、准确率高。
2. **多模态一步到位**：通义千问 Qwen-VL / 智谱 GLM-4V 直接读图出 JSON。注意现有 DeepSeek 对话 API 为纯文本、无视觉，此路线需另接视觉模型。
3. **本地 PaddleOCR + DeepSeek**：离线、数据不出本机；老 CPU 较慢、印章/手写准确率一般。

- 复用现有 `services/content-parser.js` 的 `extractTextFromFile`（docx/xlsx/txt 文本）+ `services/deepseek.js` + `extractJsonContent`。
- 复用 `AiTaskLog` 记录；`AiTaskType` 追加 `ACHIEVEMENT_OCR / ACHIEVEMENT_PARSE / PERFORMANCE_SUMMARY`。
- **降级**：无 OCR/无 key 时，老师手填，功能不受影响。
- **隐私**：扫描件含个人信息，外发云端前需告知/留策略；可对敏感件强制走本地路线。

## 六、复核工作流

状态机（对齐自评表的"自评 / 初步复核 / 最终复核"列）：

`DRAFT`（草稿）→ 教师提交 `SUBMITTED` → 部门初复 `PRELIM_REVIEWED` → 终复 `FINAL_REVIEWED`（或 `REJECTED` 退回）。

- 每条业绩带 `reviewDept`，按部门分流：资历类→组织人事处、教学类→教务处、科研类→科研处、辅导员/荣誉→学生处。
- 复核人填 `prelimScore / finalScore + reviewNote`，终复通过置 `verified=true`（前端盖"朱红印章"徽章）。
- 复用现有审批 UX 与 `flash.js` 提示。

## 九、权限模型

- 教师：仅本人档案/业绩/申报批次（沿用现有按 `teacherId` 过滤）。
- 管理员：全部 + 评分方案配置 + 复核。
- **未来**：增加"部门复核员"角色（组织人事处/教务处/科研处/学生处），按 `reviewDept` 只看本部门待复核项。v1 先由管理员统一复核。
